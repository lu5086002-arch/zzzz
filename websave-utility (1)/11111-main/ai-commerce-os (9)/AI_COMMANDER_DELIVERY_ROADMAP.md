## 1. 第一阶段：可执行开发计划

本阶段核心目标是让 AI 拥有**“眼睛”**（关联页面与全局运行时上下文）与**“精准大脑”**（回答皆须建立在业务系统的真实数据事实之上，绝不盲目瞎编）。

```
+-------------------------------------------------------------+
|                       Sprint 运行节奏                        |
|                                                             |
|   Sprint 1 (基石阶段)     Sprint 2 (场景扩充)    Sprint 3 (收尾精细化) |
|   - 极简版 AIContext       - pageType 参数扩充   - 推荐逻辑重构       |
|   - StoreKpis 获取链路     - 商品 & 订单详情解析  - 自然语言匹配扩展   |
|   - /api/ai/ask 通路打通   - 数据卡片式交互排布  - 强力 System Prompt  |
+-------------------------------------------------------------+
```

### 📅 Sprint 1：打地基（Runtime Context + StoreKpiService）
* **开发目标**：AI 能够精确、实时地回答“今天怎么样”“利润怎么样”等全店经营问题，充当“带有高精大盘数据的商业分析师”。
* **核心开发项**：
  1. **数据模型实体化**：
     * 实现后端 `AIContext` 映射（包含 `shop`、`user`、`ui` 字段）。
     * 定义并接入 `StoreKpiService.getStoreKpis(shopId)` 用于加载 GMV、客单价、退款率、纠纷比例。
  2. **调度链路跑通**：
     * 构建 `/api/ai/ask` 最简接口，仅在用户处于 Dashboard（大盘）页面并发出大盘类提问时触发。
     * 读取 `aiRuntimeStore` 生成静态上下文，合并 `StoreKpis` 输入作为提示词附件喂给 LLM。
* **交付验收标准**：
  * 在大盘页输入：“今天怎么样？” or “最近利润如何？”
  * AI 能够一字不差地拆解今日成交额、订单笔数、客单价等，且数据必须通过后端 Service 计算，修改数据库记录后再次提问，数据能即时响应。

### 📅 Sprint 2：场景扩展（商品详情 + 订单详情）
* **开发目标**：AI 进驻“详情级”页面，不需要商家解释“正在看哪个商品/哪个订单”，AI 具有定向诊断、安全风控能力。
* **核心开发项**：
  1. **路由自动化拦截**：
     * 在 `aiRuntimeStore` 的 `currentRoute` 变动时，自动解析 `/products/:id` 或 `/orders/:id`，提炼并激活 `productId`/`orderId` 指针。
  2. **单品与订单 Service 补齐**：
     * 实现 `ProductService.getProductInsight(productId, shopId)`，抓取最近 30 天销量、加购转化率、毛利空间与库存天数。
     * 实现 `OrderService.getOrderRisk(orderId, shopId)`，根据订单金额、新老客户历史表现、收货地址与欺诈判定生成风险报告。
  3. **模型意图判定扩展**：
     * 在 `/api/ai/ask` 大脑中注册商品和订单处理分支。当 `pageType === 'product_detail'` 且包含“这个商品”关键字，调用 `getProductInsight` 数据，不调用无用接口。
* **交付验收标准**：
  * **商品详情页**：输入“这个品库存储备和动销怎么样？”，AI 能结合利润率给出“该商品建议调价、面临缺货或应该套餐捆绑”的意见。
  * **订单详情页**：输入“这笔订单有异常吗？”，AI 能通过风控盾分析高危/低危并罗列出详尽的风控逻辑。

### 📅 Sprint 3：支付推荐 + 全局自然语言入口丰富
* **开发目标**：AI 可结合国家政策、商户行业及历史数据推荐支付方式，理顺所有模糊兜底命令。
* **核心开发项**：
  1. **支付网关诊断器开发**：
     * 实现 `PaymentService.getPaymentStats(shopId)`。
     * 实现 `PaymentService.getBestPaymentMethod(country, industry)`，配置策略库（如：若国家为意大利且行业为餐饮，建议启用 *Shopify Payments + PayPal + Klarna*）。
  2. **Master System Prompt 强化**：
     * 固化提示词约束模板，规定：① 先看事实数据，② 不允许捏造不存在的属性，③ 仅提供步骤和方法草案，不替用户直接写库。
* **交付验收标准**：
  * 输入：“哪个支付通道最赚钱？”
  * AI 自动抓取商家注册地与经营大类，定制化罗列 2~3 个应该在支付设置中开启的支付渠道及其原因。

## 2. 第二阶段：业务提效「Draft (草案) 机制」

> **设计准则**：AI 不再只流于舌尖，而是成为具备实操价值的“运营特助”。AI 可代为完成 80% 的工作并吐出结构化属性，保留“人类店长确认确认”作为最后 20% 的写入权限。

```
+-------------------------------------------------------------+
|                   第二阶段数据流通与拦截模型                 |
|                                                             |
|                [ 用户咨询: "给这个商品搞个促销" ]             |
|                                |                            |
|                                v                            |
|             [ AIContext 路由感知 & 调取 ProductInsight ]    |
|                                |                            |
|                                v                            |
|                [ 调用 DiscountService 决策生成 ]             |
|                                |                            |
|                                v                            |
|            [ 返回自然语言建议 + 结构化 Draft JSON Payload ]   |
|                                |                            |
|                                v                            |
|                  [ 前端渲染 "一键生成拟稿" 交互 ]             |
|                                |                            |
|           +--------------------+--------------------+       |
|           |                                         |       |
|           v (点击确认)                               v (修改) |
|  [ 调用后端真正的业务接口 ]                  [ 打开参数弹窗微调 ] |
|  [ 写入 DB 生效! ]                                            |
+-------------------------------------------------------------+
```

### 2.1 典型 Action Draft 数据契约设计

以下是后端服务应该向前端输出的“一键草稿”交互数据协议规范，通过格式化 JSON，前端可直接映射出“一键采纳并创建”按钮：

#### 契约 A：折扣草案建议数据结构 (`ActionDraft<Discount>`)
```json
{
  "actionType": "CREATE_DISCOUNT_DRAFT",
  "reason": "由于当前商品 [Sample Jacket] 的库存量 (120) 显著超出 30 天动销预期，建议降级库存风险。",
  "payload": {
    "title": "AI_Generated_Stock_Clearance",
    "type": "percentage_discount",
    "value": 20.0,
    "minPurchaseRequirement": {
      "type": "quantity",
      "value": 2
    },
    "appliesTo": {
      "collectionId": null,
      "productIds": ["gid://shopify/Product/123"]
    },
    "validFrom": "2026-06-08T08:00:00Z",
    "validTo": "2026-06-15T23:59:59Z"
  }
}
```

#### 契约 B：客户流失挽回流自动化草案 (`ActionDraft<CustomerFlow>`)
```json
{
  "actionType": "CREATE_AUTOMATION_FLOW",
  "reason": "检测到您店铺累计有 15 个客户超过 90 天未产生订单，占活跃客群比重呈上升态势，建议热部署召回链路。",
  "payload": {
    "triggerEvent": "CUSTOMER_CHURN_RISK_DETECTED",
    "delayDurationHours": 24,
    "actions": [
      {
        "type": "SEND_EMAIL",
        "template": "customer_winback_standard",
        "variables": {
          "couponCode": "COMEBACK10",
          "discountPercent": "10%"
        }
      },
      {
        "type": "ADD_TAG",
        "tag": "churn-prevention-active"
      }
    ]
  }
}
```

### 2.2 前端无缝无干扰式 Suggestion Overlay
* 不随意调整既有业务页面的基本排版布局，避免对高频熟练操作产生视觉污染。
* 在商品、订单、营销页面侧边挂载浮动的推荐区域（**Intelligence Sidebar**）。该侧栏可展示由 AI 后台通过计算主动吐出的优化草券或警报，配合一键生成草稿快捷操作。

## 4. 问题 → 工具调用决策路由矩阵

大语言模型（LLM）不能凭空推算数字。开发团队必须借助“决策路由表”，先根据用户发出的意图和输入条件解析需要调用的后端底层 API 服务，再将获取的 JSON 结果整合包装输入大模型。

| 场景意图 (`Intent`) | 典型用户问句 | 上下文依附 (`AIContext.ui`) | 后端预调用工具服务 (`Tool Services`) | 期望大模型产出方向 |
| :--- | :--- | :--- | :--- | :--- |
| **`store_overview`** | “今天店里生意怎么样？”<br>“看下我这个月的利润”<br>“汇报下运营情况” | 任意页面，若处于 `dashboard` 最佳 | `StoreKpiService.getStoreKpis(shopId)` | 提取核心指标进行涨跌幅比照，指出表现最好/拖后腿的方面，并列出今日 Top 待办事项。 |
| **`product_insight`** | “帮我看看这个商品”<br>“这个品好卖吗？”<br>“我该如何优化这个产品” | `product_detail` (绑定 `productId`) | `ProductService.getProductInsight(productId, shopId)` | 综合评价商品的动销和转化，对比其对利润的贡献度。基于库存健康状态，给出提价、多件多折、补发货的具体建议。 |
| **`order_risk`** | “这单有没有欺诈嫌疑？”<br>“订单安全风险分析”<br>“给这一单把把关” | `order_detail` (绑定 `orderId`) | `OrderService.getOrderRisk(orderId, shopId)` | 评估地址合理性、首单大额特征或渠道安全状态，确定高/中/低风险级别。输出风险理由并提供防欺诈处理指南。 |
| **`payment_recommend`**| “支付用哪个最划算？”<br>“推荐下欧洲这边的渠道配置”<br>“收欧元用什么好” | 任意页面 | `PaymentService.getBestPaymentMethod(country, industry)` | 契合店铺注册地和行业大类，推荐最佳网关搭配，陈述卡种偏好和转嫁成本优势。 |
| **`inventory_replenish`**| “有哪些品快断货了？”<br>“帮我拉一下库存预警”<br>“库存有没有风险？” | 任意页面 | `ProductService.getLowStockProducts(shopId)` | 汇总库存紧缺项目，罗列濒临告警的 SKU 清单，同时关联供应商交付周期建议合理的补货量。 |

## 5. 上线前联调与手工验收 Checklist

在产品上线前，QA 或开发负责人可以通过以下最小集验收用例，手工验证整个端到端的 Context-Aware AI Commander 是否达到了发布标准：

### 🧪 场景用例 A：大盘业绩实时研判 (在大区/仪表盘页)
* [ ] **动作 1**：在主屏幕大区搜索或 AI 指挥官窗中键入：`“最近利润和盘面怎么样”`。
* [ ] **校验 1**：观察 AI 会话输出文本。应当提取出今日/本月的销售总额、实际客单价等数据（注意检查数据应当与大盘页面看到的统计项一致）。
* [ ] **动作 2**：在后台通过手动添加等方式触发一单或数单真实的已经完成付款的新业务交易。
* [ ] **校验 2**：约 2 秒钟后再次在 AI 会议框里提问相同问题。观察输出数据，相关的销售增加额、笔数值必须发生对应更新（证明不是直接硬编码或 mock 的假数据，必须是真的在调底层 StoreKpi 接口）。

### 🧪 场景用例 B：商品动效精准解读 (在商品详情页面)
* [ ] **动作 1**：点击侧边栏“商品管理”，任选其中一个拥有真实销量的单品（记录下其内部使用的特定 ID）。
* [ ] **校验 1**：在商品详情编辑状态下发起会话意图：“这个商品该怎么卖最赚钱？”
* [ ] **校验 2**：会话内容应该自动输出：
  1. 准确解析出商品的名字；
  2. 提取最近 30 天属于该商品的销量及购买件数；
  3. AI 绝不询问“你指的是哪款商品”，而是自带眼睛自动关联页面的单品。

### 🧪 场景用例 C：收单支付精细化诊断
* [ ] **动作 1**：在后台设置中。将当前测试主站国家绑定为：`“意大利(Italy)”`，经营类别绑定为 `“服饰配饰”`。
* [ ] **动作 2**：呼出命令输入：`“帮我看看用什么收款最好”`
* [ ] **校验 2**：AI 大脑应能针对当前国家与服装品类的消费特性进行针对性推荐（建议优先开启 Google Pay、Shopify Payments 等），同时罗列推荐理由，不得给出适用于中国（如支付宝、微信支付）的离题方案。

## 6. 一页纸故事：这套 AI 系统与普通 Chatbot 的本质差异

| 衡量维度 | ❌ 传统“套壳”式 Chatbot (市面常见落后形态) |  Shopify AI OS: AI Commander (第一阶段交付姿态) |
| :--- | :--- | :--- |
| **信息获取方式** | **“瞎子摸象”**：用户必须在提问中长篇大论手动打字告诉 AI 商品名、今日销量、特定售价、纠纷率等指标。 | **“开眼决策”**：AI 自动解析当前路由。从页面当前的产品 ID、订单号到全店行业角色标签，全部静默读取。 |
| **可信度与数据真实性**| **“空口说瞎话”**：模型由于缺乏与底层系统的业务连接，在回答数字时极易出现幻觉。 | **“真枪实弹”**：AI 绝对不直接生成答案。在开口回答前，必先路由并调取后端对应的 KPI / Product 等底层微服务，获得无误的数据后方组织遣词。 |
| **操作联动能力** | **“纸上谈兵”**：只会空洞建议“您应该去做个八折大促”，用户需要自己离开会话，回到层层菜单翻寻大促入口。 | **“搭好阶梯”**：直接附带匹配规范的 `CREATE_DISCOUNT_DRAFT` 行为 Draft，商家只需轻击，即可一键呼出发起实体，不绕路。 |
| **对话范围和自律防线**| **“插科打诨”**：容易被商家绕晕，甚至会去跟商家唠闲嗑、做数学题、写垃圾代码。 | **“高冷指挥”**：任何偏离店铺生意逻辑、退单风控、支付性能之外的闲聊全数拦截，礼貌且坚定地带回后台生意正轨。 |

## 7. 演示大纲：写给高管和老板的 AI OS 商业汇报白皮书

> **目的**：帮助您迅速向公司老板、股东或业务决策方汇报，论证「AI Commander OS」的核心商业壁垒、典型痛点解决以及非聊天式智能体的必要性，展现其高昂的行业溢价价值。

### 📋 页 1：封面页 (Slide 1)
- **主标题**：🧠 **AI Commander OS：让后台自己会想的电商智能副驾**
- **副标题**：*“拒绝低效聊天，为 SaaS 商家和平台运营做业务决策，带来越富余的现金流与确定性。”*
- **核心定位**：媲美 Shopify Sidekick 级、面向全周期多任务协同的系统原生智能核心。

### 📋 页 2：痛点直面 —— 为什么传统“聊天机器人”在 SaaS 中全军覆没？ (Slide 2)
- **痛点一：脱离业务上下文** ➔ 机器人不知道商户在看哪个商品，用户解释疲惫。
- **痛点二：幻觉频出无法交易** ➔ 模型瞎编销售利润额、仓储周转率，甚至输出错误物流指导，对财务造成致命创伤。
- **痛点三：纸上谈兵无执行力** ➔ 机器人只会列出“你应该搞大促”的文字，不会编译成指令直接作用于商品、WMS及规则引擎。
- **破局**：运营后台的“智能”应该体现为**「可配置」**、**「可执行」**、**「可监控」**、**「可审计」**。

### 📋 页 3：核心价值 —— 什么是“目标权重驱动”与“多智能体博弈决策”？ (Slide 3)
- **系统核心思想**：
  商业决策从来不是单一维度的。提升 GMV 必然牺牲毛利、清理积压可能引发倾销风控。普通的 Chatbot 只有一个大脑，无法进行利益权衡。
- **AI Commander OS 决策模型**：
  - **由多个专职智能体组成博弈圆桌**：运营总监 ➔ 动态调控；定价专家 ➔ 计算让步缓冲；仓库专家 ➔ 追踪周转警戒；存量推广 ➔ 精确推荐老客。
  - **基于纳什平衡求解**：根据管理员调校的 GMV、毛利、库存与风控目标权重，在沙盘中模拟多路方案。在不损伤财务安全的边界下，获得综合指标最优解。

### 📋 页 4：场景演练 1 —— ❄️ 暖冬积货清仓战役 Playbook (Slide 4)
- **复杂商业背景**：欧洲温带异常，大批重物料防寒衣物静态积压超 120 天，面临严重的每周仓租溢耗压力。
- **AI 联动感知过程**：
  1. **读取多模态数据**：自动加载 `winter_aging_analysis.xlsx` (积滞分析) 并拼合 `temperature_anomaly.jpg` (暖冬温控气压雷达图)。
  2. **设定核心目标**：GMV 提升权重 40%，毛利率底红线 30%，周转速度天数降至 35 天内。
  3. **圆桌博弈协商**：
     * *财务智能体* 给出 35% 定价底线促销限制，严守商家底限；
     * *推广智能体* 读取加购未付客群，推荐 €10 折扣直达；
     * *供应链智能体* 拦截冬货追单，释放春装库位。
  4. **一键物理热部署**：将圆桌最终评估方案一键部署至全店、全网商户。

### 📋 页 5：场景演练 2 —— 🚨 全网多层高风险异常订单沙箱防护 (Slide 5)
- **平台高频风险**：境外洗钱欺诈、恶意羊毛党套汇、违规物流套利。
- **AI 原生联动防线**：
  - WMS 智能体与安全智能体协同：分析订单结算价格，若遭遇异常低价，在仓库打单前呼出熔断。
  - AI 指挥官直接生成过滤规则，无需人工写代码。
